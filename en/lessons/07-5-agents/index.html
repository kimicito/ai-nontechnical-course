7.5 Agent Scenarios for Procurement

⏱ 12 min · how agents automate procurement processes
How is an agent different from a language model?
LLM answers individual requests. Agent works cyclically: sets tasks itself, calls tools, gets results and decides what to do next. An agent can visit a website, download a file, send an email — without human involvement at each step.

More about agents — Level 4: Agents and Orchestration.

Scenario 1: Tender Monitoring Agent (n8n + LLM)

Task: daily check government and commercial tenders, filter by keywords, send brief digest to procurement officer.

Agent Architecture

[n8n Cron: daily at 9:00]
↓
[HTTP Request: procurement portal / customer API]
↓
[LLM Node: filter by keywords, assess relevance]
↓
[Telegram / Email: digest "5 tenders for today"]
↓
[Google Sheets: log for analytics]

What the Agent Does

  • Downloads XML/JSON from procurement platform
  • LLM reads lot descriptions, compares with procurement profile
  • Assesses relevance on 1–10 scale
  • Sends only tenders with score ≥7
  • Logs to spreadsheet for effectiveness statistics
Result: procurement officer gets ready digest instead of 2 hours of manual platform browsing. Saves 10 hours per week.

Scenario 2: Supplier Analysis Agent (OpenClaw-style Agent)

Task: upon receiving a new supplier request — automatically collect public information and prepare due diligence profile.

Agent Architecture

[System request: company name, tax ID]
↓
[OpenClaw Agent: parallel calls]
├─→ [Browser: company website → extract activity, contacts]
├─→ [API: business registry → check status, founders]
├─→ [API: credit bureau → financial indicators]
└─→ [Search: news, court cases]
↓
[On-prem LLM: consolidate into unified profile]
↓
[PDF report to folder + notification to procurement]

Environment Security

  • Public data collection — external environment (browser, API)
  • Profile consolidation, risk assessment — on-prem LLM
  • Admission decision — human (human-in-the-loop)
Important: the very fact of supplier checking and its results are internal information. Profile is prepared on-prem, even if data is collected from public sources.

Scenario 3: Commercial Proposal Comparison Agent (Hybrid)

Task: receive CPs from 3+ suppliers, structure, compare, identify hidden risks, prepare recommendation.

Agent Architecture

[Email integration: new email with subject "CP"]
↓
[Agent: extract attachments (PDF, Excel, Word)]
↓
[On-prem LLM: extract parameters from CPs]
├─ Price, terms, payment conditions
├─ Warranties, penalties, force majeure
└─ Hidden conditions (auto-renewal, sole arbitration)
↓
[LLM: comparison table + SWOT for each CP]
↓
[Notification to procurement: "CPs ready for comparison"]

Key Feature

Agent works inside the environment: receives emails, processes documents, doesn't send CP content outside. LLM analyzes conditions without seeing competitor names during comparison (if additional confidentiality is required).

Result: comparison of 5 CPs in 15 minutes instead of 3–4 hours of manual analysis. Agent doesn't miss fine print and hidden conditions.
