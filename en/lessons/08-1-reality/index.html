8.1 Reality: What AI Can Do in Construction Today

15 min · for estimators, site supervisors, engineers · capital construction

The main thesis of this lesson: an estimate is not text. LLMs don't calculate according to GESN/FER, don't know coefficients, and don't understand the resource method. If you feed a neural network an estimate as text and expect an accurate calculation — you're doomed to hallucinations.

But that doesn't mean AI is useless in construction. It is useful — but for tasks where precise normative calculation isn't needed. Let's break down where it works now and where it doesn't yet.

Why Construction Is Harder Than Procurement

In procurement, there are concrete numbers: price, lead time, quantity. An LLM compares 5 quotations — and does it well. In construction, everything is different:

What makes construction difficult for LLMs:
  • Normative databases: GESN, FER, TSN, FSSC — tens of thousands of items with coefficients
  • Resource method: cost = norm × resource price × coefficients. LLMs don't know current resource prices
  • Coefficients: Kp (profit), Knr (overhead), Kpz (loading), conversion indices — they vary by region and year
  • Local estimates: every project is unique, no two estimates are identical
  • Estimate norms ≠ market prices: GESN provides labor and machinery norms, but material prices come from the market
Example of a mistake: You upload a PDF estimate to ChatGPT and ask "check the calculation." The LLM sees text, but not what's behind the tables: which norms were applied, which coefficients, which indices. It may say "everything is correct" or find an obvious arithmetic error — but miss an estimating error of 30%.

Three Buckets of Tasks

Let's divide the construction department's tasks into three groups — by AI readiness to solve them.

🟢 Bucket 1: Quick Wins — Right Now

LLMs handle these without customization. ROI in days and weeks.

  • Normative database search: "In which GESN collection can I find bored pile construction?"
  • Decoding estimate items: "What does item FERm 01-01-001-01 mean?"
  • VOR to text conversion: "Based on the Bill of Quantities, write a method statement"
  • Text document generation: explanatory notes, method statements, letters
  • Comparing local estimates by text: "Estimate A and Estimate B — what's the difference in approach?"
  • Logic check (not calculation!): "The foundation estimate includes roofing work — is this an error?"
  • Data extraction from drawings (OCR): recognize dimensions, marks, quantities

🟡 Bucket 2: Needs Development — Months

Requires integration with estimating software, normative databases, or custom logic.

  • Automatic estimate calculation from VOR: link work volumes to GESN/FER, apply coefficients
  • Norm-based estimate validation: automatic verification: are rates selected correctly, are any works missing
  • Cost forecasting: model trained on historical data predicts final estimate from draft VOR
  • Drawing-to-estimate matching: verify that all elements on the drawing are accounted for in the estimate
  • VOR extraction from BIM models: automatic quantity takeoff from 3D models

🔴 Bucket 3: Needs Agents — Years

Requires a multi-agent system with access to external data and autonomous decision-making.

  • Full estimating automation: from drawing to approved estimate without human intervention
  • Automatic resource price monitoring: an agent monitors the market and updates the estimate in real time
  • Construction risk forecasting: an agent analyzes weather, suppliers, subcontractors, and warns of delays
  • Automatic change management: when the project changes, an agent recalculates everything: estimate, schedule, resources

Where to Pilot: Bucket 1

Don't start with automatic estimate calculation — that's Bucket 2, it needs development. Start with tasks that deliver results within a week.

Step 1: Normative Database Search

Example Prompt
I need to find a norm for constructing a monolithic reinforced concrete strip foundation.

Questions:
1. In which GESN or FER collection should I look?
2. Which rates are suitable for a foundation 1.5 m deep and 0.4 m wide?
3. What resources are included in the norm (concrete, rebar, formwork, labor)?
4. What coefficients are typically applied to this norm?

The LLM won't give an exact calculation, but it will point you in the right direction — saving 30 minutes of searching through collections.

Step 2: Decoding Estimate Items

Example Prompt
Here is a list of items from a local estimate:
- FERm 08-02-001-01
- GESNm 81-05-001-01
- TSN-2001-08-02-001

Decode each item:
1. Full work name
2. Unit of measurement
3. Main resources (materials, labor, machinery)
4. Typical area of application

Step 3: Logic Check of an Estimate

Example Prompt
I have uploaded a PDF of a local estimate for building capital repairs.

Check the logic (DO NOT calculate sums):
1. Do all estimate sections match the types of work in the explanatory note?
2. Are there any works that shouldn't be in this type of repair?
3. Are there any missing works typical for capital repairs?
4. Does the sequence of works match the technology?

Provide a list of suspicious items with explanations.

Step 4: Generating Explanatory Notes

Example Prompt
Based on the local estimate (PDF attached), draft an explanatory note:

1. General provisions (object, client, contractor)
2. Basis for estimate preparation
3. List of applied rates and norms
4. Features of coefficient application
5. Total cost and work duration

Format: official, per GOST R 21.1101-2013.

What NOT to Do with LLMs in Construction

Red Flags — Tasks Where LLMs Will Mislead You:
  • Calculating an estimate "from scratch" from text: LLMs don't know current resource prices, indices, and coefficients
  • Checking arithmetic in complex estimates: may miss an error in a coefficient but "confirm" the total
  • Selecting norms by work description: LLM may suggest a similar but incorrect rate
  • Using for estimate justification in court or expertise: LLM conclusions have no legal force
Safety Rule: An LLM is an estimator's assistant, not a replacement. It speeds up routine (search, text, logic checking), but accurate calculation still belongs to humans and estimating software (GRAND-Estimate, A0, Smeta.ru).

Roadmap: From Bucket 1 to Bucket 2

When Bucket 1 starts delivering results, you can move further:

  1. Month 1–2: all estimators use LLMs for norm search, item decoding, text generation
  2. Month 3–4: integration with GRAND-Estimate or A0 API for automatic item export
  3. Month 5–6: custom tool: upload VOR → auto-select norms → draft estimate (requires development)
  4. Month 7–12: train a model on company historical estimates for cost forecasting
