7.5 Агентские сценарии для Закупок

⏱ 12 мин · как агенты автоматизируют закупочные процессы
Чем агент отличается от языковой модели?
LLM отвечает на одиночные запросы. Агент работает циклически: сам ставит задачи, вызывает инструменты, получает результат и решает, что делать дальше. Агент может зайти на сайт, скачать файл, отправить письмо — без участия человека в каждом шаге.

Подробнее об агентах — Уровень 4: Агенты и оркестрация.

Сценарий 1: Агент мониторинга тендеров (n8n + LLM)

Задача: ежедневно проверять госзакупки и коммерческие тендеры, фильтровать по ключевым словам, присылать короткий дайджест закупщику.

Архитектура агента

[n8n Cron: ежедневно в 9:00]
↓
[HTTP Request: zakupki.gov.ru / API заказчика]
↓
[LLM Node: фильтрация по ключевым словам, оценка релевантности]
↓
[Telegram / Email: дайджест «5 тендеров на сегодня»]
↓
[Google Sheets: запись в реестр для аналитики]

Что делает агент

  • Скачивает XML/JSON с площадки госзакупок
  • LLM читает описания лотов, сравнивает с профилем закупщика
  • Оценивает релевантность по шкале 1–10
  • Отправляет только тендеры с оценкой ≥7
  • Записывает в таблицу для статистики эффективности
Результат: закупщик получает готовый дайджест вместо 2 часов ручного просмотра площадок. Экономия 10 часов в неделю.

Сценарий 2: Агент анализа поставщиков (OpenClaw-стиль агент)

Задача: при поступлении заявки на нового поставщика — автоматически собрать публичную информацию и подготовить due diligence-профиль.

Архитектура агента

[Заявка в систему: название компании, ИНН]
↓
[Агент OpenClaw: параллельные вызовы]
├─→ [Браузер: сайт компании → извлечь деятельность, контакты]
├─→ [API: ЕГРЮЛ → проверить статус, учредителей]
├─→ [API: СПАРК/Контур → финансовые показатели]
└─→ [Поиск: новости, судебные дела]
↓
[LLM On-prem: сведение в единый профиль]
↓
[PDF-отчёт в папку + уведомление закупщику]

Безопасность контура

  • Публичный сбор данных — внешний контур (браузер, API)
  • Сведение профиля, оценка рисков — on-prem LLM
  • Решение о допуске — человек (human-in-the-loop)
Важно: сам факт проверки поставщика и её результаты — внутренняя информация. Профиль готовится на on-prem, даже если данные собраны из публичных источников.

Сценарий 3: Агент сравнения коммерческих предложений (гибрид)

Задача: получить КП от 3+ поставщиков, структурировать, сравнить, выявить скрытые риски, подготовить рекомендацию.

Архитектура агента

[Email-интеграция: новое письмо с темой «КП»]
↓
[Агент: извлечение вложений (PDF, Excel, Word)]
↓
[On-prem LLM: извлечение параметров из КП]
├─ Цена, сроки, условия оплаты
├─ Гарантии, штрафы, форс-мажор
└─ Скрытые условия (автопролонгация, единоличный арбитраж)
↓
[LLM: сравнительная таблица + SWOT каждого КП]
↓
[Уведомление закупщику: «КП готовы к сравнению»]

Ключевая особенность

Агент работает внутри контура: получает письма, обрабатывает документы, не отправляет содержимое КП наружу. LLM анализирует условия, не видя названий конкурентов при сравнении (если требуется дополнительная конфиденциальность).

Результат: сравнение 5 КП за 15 минут вместо 3–4 часов ручного разбора. Агент не пропускает мелкий шрифт и скрытые условия.

Сравнение подходов

ПодходКогда использоватьСложность
n8n + LLM APIПростые сценарии: парсинг, фильтрация, уведомленияНизкая
