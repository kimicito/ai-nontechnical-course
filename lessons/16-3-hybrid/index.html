Уровень 16 — Комплаенс

16.3 Гибридные задачи

⏱ 20 мин · для комплаенс-офицеров
Гибридный подход
Общую методологию, структуру, типовую логику готовим во внешнем ИИ.
Конкретные имена, сделки, транзакции, стратегические решения — в on-prem.

Принцип гибридного подхода для комплаенса

Комплаенс сочетает чувствительность нескольких функций. Гибридный подход позволяет использовать внешний ИИ для методологии и общей логики, сохраняя конфиденциальные данные внутри.

Правила обезличивания для облака:
Что обязательно убратьЧто можно оставить
Названия компаний-контрагентовТип контрагента (поставщик/покупатель/агент)
Конкретные суммы транзакцийМетодология анализа паттернов
Имена бенефициаров/PEPТип риска (PEP/санкции/коррупция)
Номера счетов, даты сделокТипичные сроки в отрасли
Названия подразделений компанииТип процесса (закупки/продажи/наём)
Стратегические решения советаОбщие принципы governance

1. Санкционный скрининг (гибрид)

Сам скрининг — через специализированные системы (не genAI). Интерпретация результатов — с обезличенными данными во внешнем ИИ, финальное решение — в on-prem.

Шаг 1: Облако — интерпретация на обезличенных данных
Проанализируй сценарий санкционного скрининга:

Сценарий (обезличенный):
- Тип контрагента: физическое лицо
- Страна: [УКАЖИТЕ РЕГИОН без конкретной страны если чувствительно]
- Алерт: частичное совпадение имени с фигурантом списка OFAC
- Дополнительные данные: дата рождения не совпадает, страна другая

Сделай:
1. Анализ степени совпадения (критерии OFAC)
2. Вероятность ложного срабатывания
3. Дополнительные проверки, которые нужно провести
4. Методология принятия решения (да/нет/дополнительная проверка)
5. Документальный след (что зафиксировать)

Важно: без конкретного имени, только методология.
Шаг 2: On-prem — финальное решение
Прими решение по конкретному контрагенту:

Контрагент: [УКАЖИТЕ ИМЯ]
Результаты скрининга: [ВСТАВЬТЕ]
Методология (из облака): [ВСТАВЬТЕ]

Дополнительные внутренние данные:
- История сотрудничества: [УКАЖИТЕ]
- Бизнес-обоснование: [УКАЖИТЕ]
- Риск отказа: [УКАЖИТЕ]

Прими решение:
1. Одобрить / Одобрить с условиями / Отказать
2. Обоснование (с привязкой к конкретным фактам)
3. Условия (если применимо)
4. Документальное оформление
5. Срок пересмотра

Экономия: методология в облаке (1 час) + решение в on-prem (1 час).

2. Комплаенс риск-оценка (гибрид)

Общую методологию и матрицу рисков готовим снаружи, применение к реальным бизнес-единицам — внутри.

Шаг 1: Облако — методология и матрица
Разработай методологию комплаенс риск-оценки:

Параметры (общие):
- Отрасль: [УКАЖИТЕ ТИП без названия компании]
- География операций: [УКАЖИТЕ РЕГИОНЫ]
- Типы контрагентов: [УКАЖИТЕ: госучреждения/частные/агенты]
- Продукты/услуги: [УКАЖИТЕ ТИП]

Сделай:
1. Матрица рисков (факторы: страна, контрагент, продукт, канал)
2. Методология оценки (веса факторов)
3. Категории риска (высокий/средний/низкий)
4. Меры контроля для каждой категории
5. Частота пересмотра
6. Шаблон оценки (таблица)

Важно: методология без привязки к конкретной компании.
Шаг 2: On-prem — применение к реальным процессам
Примени методологию к реальным бизнес-процессам:

Методология (из облака): [ВСТАВЬТЕ]

Бизнес-процессы компании:
- Процесс 1: [УКАЖИТЕ НАЗВАНИЕ] — страна [УКАЖИТЕ], контрагенты [УКАЖИТЕ]
- Процесс 2: [УКАЖИТЕ НАЗВАНИЕ] — страна [УКАЖИТЕ], контрагенты [УКАЖИТЕ]
- Процесс 3: [УКАЖИТЕ НАЗВАНИЕ] — страна [УКАЖИТЕ], контрагенты [УКАЖИТЕ]

Сделай:
1. Оценка риска каждого процесса
2. Карта уязвимостей
3. Приоритеты усиления контроля
4. План мероприятий (кто, что, когда)
5. Сроки пересмотра

Экономия: методология в облаке (2 часа) + применение в on-prem (2 часа).

3. Сценарии тренингов на основе реальных кейсов (гибрид)

Структуру реального кейса обезличиваем и прорабатываем снаружи, полностью реальный кейс — внутри.

Шаг 1: Облако — обезличенный кейс
Разработай сценарий для комплаенс-тренинга:

Кейс (обезличенный):
- Ситуация: сотрудник отдела [УКАЖИТЕ ФУНКЦИЮ] получил предложение о "подарке" от потенциального контрагента
- Контрагент: компания из страны с [УКАЖИТЕ УРОВЕНЬ коррупционного риска]
- Сумма "подарка": [УКАЖИТЕ ДИАПАЗОН без точной суммы]
- Контекст: компания ведёт переговоры о [УКАЖИТЕ ТИП СДЕЛКИ]

Сделай:
1. Описание ситуации для тренинга
2. Ключевые вопросы для обсуждения
3. Правильные и неправильные действия
4. Правовые последствия
5. Порядок действий сотрудника
6. Роль руководителя

Важно: без названий, дат, сумм.
Шаг 2: On-prem — реальный кейс для внутреннего использования
Адаптируй сценарий под реальный инцидент:

Реальный кейс (внутренний):
[УКАЖИТЕ ФАКТЫ ИНЦИДЕНТА]

Сценарий (из облака): [ВСТАВЬТЕ]

Сделай:
1. Адаптированный сценарий с реальными деталями
2. Дополнительные нюансы (специфика компании)
3. Ссылки на внутренние политики
4. Контакты для обращения
5. Форма фиксации решения

Важно: только для внутреннего использования, не для внешних тренингов.

Экономия: обезличенный кейс в облаке (1.5 часа) + реальный адаптация в on-prem (1 час).

4. Due diligence контрагента (гибрид)

Публичная часть (открытые реестры, суды, СМИ) — через внешний ИИ как ресёрч. Сведение с внутренней оценкой риска — в on-prem.

Шаг 1: Облако — публичный ресёрч
Проведи публичный due diligence типового контрагента:

Тип контрагента: [УКАЖИТЕ: агент/посредник/поставщик госучреждений]
Страна: [УКАЖИТЕ]
Отрасль: [УКАЖИТЕ]

Публичные источники:
1. Реестры юридических лиц
2. Судебные базы
3. СМИ и новости
4. Санкционные списки
5. PEP-базы (публичные)

Сделай:
1. Чек-лист публичных проверок
2. Типовые red flags по категории контрагента
3. Методология оценки риска
4. Шаблон отчёта о публичной части

Важно: без конкретного названия компании.
Шаг 2: On-prem — внутренняя оценка
Сведи результаты с внутренней оценкой:

Контрагент: [УКАЖИТЕ НАЗВАНИЕ]
Публичные данные (из облака): [ВСТАВЬТЕ]

Внутренние данные:
- История сотрудничества: [УКАЖИТЕ]
- Объём сделок: [УКАЖИТЕ]
- Связанные лица: [УКАЖИТЕ]
- Внутренние риски: [УКАЖИТЕ]

Сделай:
1. Сводная таблица рисков
2. Итоговая оценка (одобрить/условия/отказать)
3. Рекомендуемые условия
4. План мониторинга

Экономия: публичная часть в облаке (1.5 часа) + внутренняя оценка в on-prem (1 час).

Проверка знаний

Вопрос 1: Почему санкционный скрининг — гибрид, а не полностью в облаке?

Потому что методология публична, а решение по конкретному имени — конфиденциально
Потому что скрининг требует специализированного ПО
Можно полностью в облаке, если обезличить

Вопрос 2: Комплаенс риск-оценка — что в on-prem?

Применение к реальным бизнес-процессам и подразделениям
Разработка матрицы рисков
Определение весов факторов риска

Вопрос 3: Сценарии тренингов — зачем обезличивать?

Чтобы не раскрыть реальные инциденты и вовлечённых лиц
Чтобы страховая компания не узнала
Чтобы сотрудники не узнали, что это реальный кейс
