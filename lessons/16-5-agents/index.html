Уровень 16 — Комплаенс

16.5 Агентские сценарии для Комплаенса

⏱ 12 мин · санкции, AML, whistleblower, расследования
Чем агент отличается от языковой модели?
LLM анализирует по запросу. Агент работает постоянно: мониторит санкционные списки, анализирует транзакции, обрабатывает обращения — без команды «проверь сейчас». Подробнее — Уровень 4.

Сценарий 1: Агент мониторинга санкций (n8n + API)

Задача: отслеживать изменения санкционных списков, проверять контрагентов, алертить при попадании.

[Cron: каждые 4 часа]
↓
[API: запрос обновлений санкционных списков]
├─→ OFAC SDN
├─→ ЕС санкции
├─→ ООН
├─→ РФ (компенсационный список)
└─→ Национальные списки (КНР, ОАЭ и др.)
↓
[LLM: анализ изменений]
├─→ Новые лица/компании
├─→ Изменения существующих записей
└─→ Отмены санкций
↓
[Автоматическая проверка контрагентов компании]
├─→ Сверка с базой поставщиков/клиентов
├─→ Сверка с базой бенефициаров
└─→ Проверка транзитных цепочек
↓
[При совпадении: критический алерт]
├─→ Email CCO + юристу
├─→ Блокировка операций (интеграция с ERP)
└─→ Запуск процедуры выхода
↓
[Ежедневный отчёт: изменения, проверки, риски]
Результат: реакция на санкционные изменения за часы, а не дни. Минимизация риска « inadvertent violation».

Сценарий 2: Агент анализа транзакций (OpenClaw-агент)

Задача: анализировать алерты AML, формировать SAR/STR, вести документацию.

[Триггер: алерт AML-системы (подозрительная транзакция)]
↓
[Агент: сбор контекста]
├─→ История транзакций контрагента
├─→ География (рисковые юрисдикции)
├─→ Сумма vs профиль клиента
├─→ Время (нерегламентированное)
└─→ Связанные лица
↓
[On-prem LLM: анализ обоснованности]
├─→ Ложный алерт: причины
├─→ Требует расследования: вопросы
└─→ Критический: рекомендация SAR/STR
↓
[При подтверждении: автоматическое формирование SAR/STR]
├─→ Структура по требованиям регулятора
├─→ Описание транзакции
├─→ Обоснование подозрения
└─→ Приложения
↓
[Уведомление комплаенс-офицеру + запись в журнал]

Сценарий 3: Агент whistleblower (гибридный)

Задача: обрабатывать обращения, классифицировать, маршрутизировать, обеспечивать конфиденциальность.

[Поступление обращения (формы, email, телефон)]
↓
[Агент: первичная обработка]
├─→ Извлечение текста (транскрипция если аудио)
├─→ Классификация: коррупция / харассмент / безопасность / финансы / другое
├─→ Оценка срочности
└─→ Проверка на дубли (похожие обращения)
↓
[On-prem LLM: обезличивание + формирование карточки]
├─→ Удаление идентификаторов заявителя (если не анонимно)
├─→ Структурирование фактов
└─→ Формирование вопросов для расследования
↓
[Маршрутизация]
├─→ Коррупция → комитет по этике
├─→ Харассмент → HR + юрист
├─→ Финансы → внутренний аудит
└─→ Безопасность → служба безопасности
↓
[Создание задачи с дедлайном]
↓
[Мониторинг: просрочка → эскалация на совет директоров]
Критично: вся обработка whistleblower — on-prem. Данные заявителя не покидают контур. Обезличивание — обязательно.
