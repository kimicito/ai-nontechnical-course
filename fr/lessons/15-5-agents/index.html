Niveau 15 — Fonction juridique

15.5 Scénarios agents pour la fonction juridique

⏱ 20 min · agents pour la surveillance législative, la vérification des contrats, la due diligence
Agent vs LLM
Un LLM répond à une question isolée. Un agent effectue une séquence d'actions : surveille, compare, alerte, crée des tâches. Pour les scénarios agents juridiques, voir Niveau 4 — Qu'est-ce qu'un agent.

Scénario 1 : Agent de surveillance législative (n8n/no-code)

Problème : les changements législatifs dans les domaines du travail, des sociétés, de la fiscalité affectent l'entreprise, mais les juristes n'ont pas le temps de surveiller quotidiennement les publications.

Architecture de l'agent

Flux de données
1. Sources : sites des parlements, registres officiels, bases de données juridiques
2. Déclencheur : publication d'un nouveau document (webhook/API)
3. Filtrage : pertinence pour l'entreprise (secteur, géographie)
4. Analyse LLM : impact sur l'entreprise (général, sans données internes)
5. Action : création d'une tâche dans le système juridique
6. Notification : alerte au juriste responsable

Données sensibles : la liste des zones d'intérêt de l'entreprise (secteur, géographie, types de contrats) — information interne. Utilisez on-prem pour le stockage et le traitement des conclusions.

Scénario 2 : Agent de vérification des contrats (full AI agent)

Problème : le flux de contrats entrants est élevé, la vérification manuelle prend trop de temps.

Architecture de l'agent

Flux de données
1. Déclencheur : nouveau contrat dans le système de gestion documentaire
2. Extraction : texte du contrat (OCR si nécessaire)
3. Comparaison LLM : écarts par rapport au standard de l'entreprise
4. Évaluation : risque de chaque écart (élevé/moyen/faible)
5. Action : génération d'amendements proposés
6. Notification : tâche pour le juriste avec priorité

Données sensibles : le texte complet du contrat, le standard de l'entreprise, les conditions commerciales — uniquement on-prem. L'agent doit fonctionner dans le contour interne.

Scénario 3 : Agent de due diligence (hybride)

Problème : la due diligence des contreparties nécessite la collecte de données publiques et leur comparaison avec les données internes.

Architecture de l'agent

Flux de données
1. Déclencheur : nouvelle transaction/nouveau contrepartie
2. Partie publique (cloud) :
   - Recherche dans les registres ouverts
   - Vérification des listes de sanctions publiques
   - Analyse des médias ouverts
3. Partie interne (on-prem) :
   - Comparaison avec l'historique des relations
   - Évaluation du risque interne
   - Décision finale
4. Action : rapport de due diligence dans le système interne

Sécurité : la partie publique utilise des sources ouvertes et peut être externalisée. La partie interne — uniquement on-prem.

Vérification des connaissances

Question 1 : Pourquoi l'agent de vérification des contrats — uniquement on-prem ?

Parce que les contrats contiennent des conditions commerciales confidentielles
On peut anonymiser les contrats pour le cloud
Parce que la vérification nécessite une précision maximale

Question 2 : L'agent de surveillance législative — quel contour pour les sources ?

Cloud — les lois et publications sont publiques
On-prem — les sources peuvent contenir des données internes
Hybride — sources à l'extérieur, conclusion à l'intérieur
